요청 1건이 거치는 통제 지점

거버넌스 · Governance

누가 · 무엇을 · 얼마나 · 어떻게 통제되는가

🪪 신원 (Identity)

IAM Identity Center(SSO)가 신뢰 기준. AWSReservedSSO_ ARN만 허용, 직접 IAM 롤 우회 차단(403)

🔏 인가 (AuthZ)

API Gateway AWS_IAM(SigV4). 권한셋에 execute-api:Invoke만 부여 — 최소 권한

🔑 키 라이프사이클

사용자별 가상키 발급·DynamoDB 캐시(TTL). master key는 Secrets Manager grantRead. 장기 키 배포 없음

📊 티어링 · 예산

권한셋→LiteLLM 팀. economy=모델 allowlist + 예산 캡($50). standard=전 모델

🛡️ 정책 (Policy)

Bedrock Guardrail + 시크릿 탐지가 모든 Claude 요청에 default 적용. MCP는 키별 access group으로 스코프

🧾 감사 (Audit)

키별 사용량·예산, Langfuse 프롬프트/트레이스, CloudWatch 지표. cdk-nag 억제는 사유 명시

데이터 레지던시 · 네트워크 격리

데이터는 어디에 머무는가

📍 데이터 위치 (단일 리전 고정)

  • 대화/키/사용량 → Aurora Serverless v2 (격리 서브넷, 저장 암호화) — 배포 리전 내
  • 가상키 캐시 → DynamoDB (배포 리전, PITR)
  • 시크릿 → Secrets Manager (배포 리전)
  • 프롬프트/트레이스 → Langfuse → 동일 Aurora (리전 내)
  • 모델 추론 → Bedrock (선택 리전 inference profile)
  • CloudFront → CACHING_DISABLED — 본문을 엣지에 캐시하지 않음

🔒 격리 보장

  • 인터넷 미경유 — Bedrock·Secrets·SSM·ECR·Logs·AgentCore를 VPC 인터페이스 엔드포인트로 호출
  • ALB 비공개 — internal-only, CloudFront VPC Origin만 진입
  • DB 격리 — Aurora는 isolated 서브넷(egress 없음), SG로 LiteLLM/λ만 5432 허용
  • 리전 고정 — 메인 스택 단일 리전(CDN만 us-east-1 · CloudFront 요구사항)
  • S3/DynamoDB — Gateway 엔드포인트(무료, 리전 내)

데이터 레지던시 요건이 있으면 배포 리전을 고정하고 모든 영속 데이터가 그 리전에 머무름을 위 경로로 보장

가드레일 · Defense in Depth

요청마다 3계층 방어

Layer 1 · 시크릿 탐지

LiteLLM hide-secrets (pre-call) — API 키/토큰 패턴을 요청 단계에서 마스킹·차단. 모든 모델 공통

Layer 2 · Bedrock Guardrail

콘텐츠: HATE·INSULTS·SEXUAL·VIOLENCE·MISCONDUCT (HIGH)

PII 차단: 카드·SSN·AWS Access/Secret Key + API키 정규식

금지 토픽: 해킹툴·데이터 유출

Layer 3 · 도구(MCP) 스코프

가상키별 mcp_access_groups(예 default_tools). 팀 멤버십 없는 키는 도구 사용 불가 — 검색 트래픽도 게이트웨이 경유

⚠️ Bedrock Guardrail은 bedrock-runtime(Claude) 전용 → GPT(Mantle) 경로는 Layer 1로 보완하고 정책 공백을 문서화(constraints.md)

🪪 IAM Identity Center (SSO) — SSO 경로 필수

  • Token Service는 AWSReservedSSO_ 신원만 허용 — 비-SSO 우회 경로 없음(설계)

  • 자격 소스: 내장 디렉터리 / 외부 IdP(Okta·Entra)

🔐 Admin UI 로그인

  • https://<cloudfront-domain>/ui/
  • 자격증명은 master key 단 하나 — Secrets Manager에서 조회(CDK 출력·로그에 절대 노출 안 함)
  • developer는 master key 불필요 — SSO 가상키만 사용

👥 팀·사용자 생성 (SSO 매핑)

  • 자동 경로(기본): SSO 권한세트 로그인 → Token Lambda가 권한세트명과 동일한 team_alias로 lookup-or-create (1:1 무분기 매칭)
  • 수동 경로: UI Teams → + New Team 또는 POST /team/new로 선점 생성 — team_alias를 권한세트명과 동일하게 입력
  • team_alias는 권한세트명 자체와 정확히 일치해야 함(Lambda 상수 아님) — 오타 시 의도한 팀이 아닌 별도 팀으로 조용히 연결됨

✅ 관리자가 알아야 할 결론

이 스킬로 생성된 게이트웨이는 기본값부터 권한세트명 = LiteLLM team_alias로 1:1 매칭되도록 코드가 생성됩니다. 그래서 새 조직·팀이 생겨도 개발자 개입·코드 수정·재배포 없이, IdC 콘솔 + LiteLLM Admin UI 클릭만으로 온보딩·예산변경·모델 제한이 끝납니다.

활성화·확인

🔧 권한세트명과 team명이 같아야 하는 이유

이 스킬이 생성하는 Token Lambda(_resolve_team_id)는 권한세트명을 그대로 LiteLLM team_alias로 사용합니다 — 별도의 매핑 테이블이나 if/else 분기가 없습니다. 그래서 3번(권한세트)과 4번(LiteLLM 팀)의 이름을 다르게 지으면 그 그룹의 로그인은 엉뚱한(또는 새로) 팀으로 연결되어 의도한 예산·모델 제한이 적용되지 않습니다. 이름을 똑같이 맞추는 것 자체가 유일한 연결 고리이므로, 온보딩 때마다 3·4번의 이름 일치만 지키면 코드를 열어볼 일이 없습니다.

잊지 말 것: 4번을 건너뛰면 조용히 default team으로 빠져 예산·allowlist가 적용 안 됨 · 근거: shared/patterns/lambda-handlers.md · shared/reference/sso-setup.md · shared/reference/litellm-admin-guide.md §2
