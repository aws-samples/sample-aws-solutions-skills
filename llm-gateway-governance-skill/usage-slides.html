거버넌스 · Governance

누가 · 무엇을 · 얼마나 · 어떻게 통제되는가

🪪 신원 (Identity)

두 모드: org-sso(IdC 조직 인스턴스, AWSReservedSSO_ ARN만 허용) 또는 cognito-native(계정 인스턴스/조직 SSO 불가 시 — Cognito User Pool이 유일 신원원, SAML/IdC 페더레이션 없음)

🔏 인가 (AuthZ)

org-sso: API GW AWS_IAM(SigV4), 권한셋 execute-api:Invoke만. cognito-native: API GW Cognito authorizer(access token만, id_token 401), cognito:groups→팀

🔑 키 라이프사이클

사용자별 가상키 발급·DynamoDB 캐시(TTL). master key는 Secrets Manager grantRead. 장기 키 배포 없음

📊 티어링 · 예산

권한셋→LiteLLM 팀. economy=모델 allowlist + 예산 캡($50). standard=전 모델

🛡️ 정책 (Policy)

Bedrock Guardrail + 시크릿 탐지가 모든 Claude 요청에 default 적용. MCP는 키별 access group으로 스코프

🧾 감사 (Audit)

키별 사용량·예산, Langfuse 프롬프트/트레이스, CloudWatch 사용량 대시보드(모델·팀·사용자별 토큰/비용/지연 — EMF 자동 수집). cdk-nag 억제는 사유 명시

데이터 레지던시 · 네트워크 격리

데이터는 어디에 머무는가

📍 데이터 위치 (단일 리전 고정)

  • 대화/키/사용량 → Aurora Serverless v2 (격리 서브넷, 저장 암호화) — 배포 리전 내
  • 가상키 캐시 → DynamoDB (배포 리전, PITR)
  • 시크릿 → Secrets Manager (배포 리전)
  • 프롬프트/트레이스 → Langfuse → 동일 Aurora (리전 내)
  • 모델 추론 → Bedrock (선택 리전 inference profile)
  • ALB idleTimeout → 기본 900s(최대 4000s) — 장시간 완성 흡수(CloudFront 120s 상한 제거)

🔒 격리 보장

  • 인터넷 미경유 — Bedrock·Secrets·SSM·ECR·Logs·AgentCore를 VPC 인터페이스 엔드포인트로 호출
  • 공개 ALB 엣지 — 두 모드 모두 인터넷 페이싱(acm=HTTPS:443, http=HTTP:80) · SG CIDR allowlist 제한, 내부 ALB는 Token Service 전용
  • DB 격리 — Aurora는 isolated 서브넷(egress 없음), SG로 LiteLLM/λ만 5432 허용
  • 리전 고정 — 메인 스택 단일 리전(AgentCore/Mantle만 us-east-1)
  • S3/DynamoDB — Gateway 엔드포인트(무료, 리전 내)

데이터 레지던시 요건이 있으면 배포 리전을 고정하고 모든 영속 데이터가 그 리전에 머무름을 위 경로로 보장

가드레일 · Defense in Depth

요청마다 3계층 방어

Layer 1 · 시크릿 탐지

LiteLLM hide-secrets (pre-call) — API 키/토큰 패턴을 요청 단계에서 마스킹·차단. 모든 모델 공통

Layer 2 · Bedrock Guardrail

콘텐츠: HATE·INSULTS·SEXUAL·VIOLENCE·MISCONDUCT (HIGH)

PII 차단: 카드·SSN·AWS Access/Secret Key + API키 정규식

금지 토픽: 해킹툴·데이터 유출

Layer 3 · 도구(MCP) 스코프

가상키별 mcp_access_groups(예 default_tools). 팀 멤버십 없는 키는 도구 사용 불가 — 검색 트래픽도 게이트웨이 경유

⚠️ Bedrock Guardrail은 bedrock-runtime(Claude) 전용 → GPT(Mantle) 경로는 Layer 1로 보완하고 정책 공백을 문서화(constraints.md)

배포 후 운영 · org-sso 사용자·그룹 → LiteLLM 권한/예산

org-sso 온보딩 — IdC + Admin UI만으로 반복 가능

✅ 관리자가 알아야 할 결론

이 스킬로 생성된 게이트웨이는 기본값부터 권한세트명 = LiteLLM team_alias로 1:1 매칭되도록 코드가 생성됩니다. 그래서 새 조직·팀이 생겨도 개발자 개입·코드 수정·재배포 없이, IdC 콘솔 + LiteLLM Admin UI 클릭만으로 온보딩·예산변경·모델 제한이 끝납니다.

1
IdC에서 사용자 추가
IAM Identity Center 콘솔 → Users → Add user — 이미 있는 사용자면 스킵
사용자 생성
2
IdC에서 그룹 추가 + 사용자 배정
IdC 콘솔 → Groups → Create group (팀명 그대로, 예: team-research) → Add users to group
그룹 구성
3
권한세트 생성 + 계정 할당
IdC 콘솔 → Permission sets → Create, 그룹과 동일한 이름(밑줄 금지) → execute-api:Invoke-only 정책 → AWS accounts에서 그룹에 배정
IAM 매핑
4
LiteLLM Admin UI에서 팀 생성 + 예산·모델 설정
Admin UI → Teams → + New Team, team_alias = 3번과 동일한 이름 → Models(allowlist), Max Budget/Budget Duration을 화면에서 직접 입력 — 코드/배포 없음
권한·예산 확정
5
비밀번호 활성화 + 검증
IdC 콘솔(전용) → Reset password → 사용자가 aws sso login 후 첫 호출 시 Admin UI Teams에서 반영 확인
활성화·확인

🔧 권한세트명과 team명이 같아야 하는 이유

이 스킬이 생성하는 Token Lambda(_resolve_team_id)는 권한세트명을 그대로 LiteLLM team_alias로 사용합니다 — 별도의 매핑 테이블이나 if/else 분기가 없습니다. 그래서 3번(권한세트)과 4번(LiteLLM 팀)의 이름을 다르게 지으면 그 그룹의 로그인은 엉뚱한(또는 새로) 팀으로 연결되어 의도한 예산·모델 제한이 적용되지 않습니다. 이름을 똑같이 맞추는 것 자체가 유일한 연결 고리이므로, 온보딩 때마다 3·4번의 이름 일치만 지키면 코드를 열어볼 일이 없습니다.

잊지 말 것: 4번을 건너뛰면 조용히 default team으로 빠져 예산·allowlist가 적용 안 됨 · 근거: shared/patterns/lambda-handlers.md · shared/reference/sso-setup.md · shared/reference/litellm-admin-guide.md §2

배포 후 운영 · cognito-native 사용자·그룹 → LiteLLM 권한/예산

cognito-native 온보딩 — Cognito + Admin UI만으로 (조직 SSO 불필요)

✅ 관리자가 알아야 할 결론

조직 SSO를 못 쓰는 환경(IdC 계정 인스턴스 등)에서도 Cognito User Pool Group명 = LiteLLM team_alias로 1:1 매칭되도록 코드가 생성됩니다. IdC·permission set 없이, Cognito 콘솔·CLI + LiteLLM Admin UI 클릭만으로 온보딩·예산변경·모델 제한이 끝납니다.

1
Cognito 사용자 생성
Cognito가 유일 신원원(self-signup off) — 첫 사용자는 배포 시 에이전트가 자동 생성(Discovery 답변 기반, 풀은 0명으로 배포됨). 추가는 aws cognito-idp admin-create-user --user-pool-id <pool> --username dev@ex.com (또는 콘솔)
사용자 생성
2
User Pool Group 생성 + 사용자 추가
Group명 = 팀(teamGroupPrefix 접두어, 예 llmgw-team-research) → admin-add-user-to-group. IdC permission set 불필요
그룹 구성
3
cognito:groups 자동 반영
로그인 시 access token에 그룹이 실려 Token Lambda가 teamGroupPrefix로 필터해 팀 결정 — 별도 IAM·Identity Store 호출 없음
신원→팀
4
LiteLLM Admin UI에서 팀 생성 + 예산·모델 설정
Admin UI → Teams → + New Team, team_alias = 2번 Group명과 동일 → Models(allowlist)·Max Budget을 화면에서 직접 입력 — 코드/배포 없음
권한·예산 확정
5
비밀번호 설정 + 검증
admin-set-user-password --permanent(또는 첫 로그인 강제변경) → 사용자가 llmgw-login(Cognito Hosted UI) 후 Admin UI Teams에서 반영 확인
활성화·확인

🔧 Group명 = team_alias가 같아야 하는 이유

Token Lambda(_resolve_team_id)는 검증된 cognito:groups 중 teamGroupPrefix에 맞는 하나를 그대로 team_alias로 씁니다(require-single-team-group: 0개·다중이면 거부). Identity Store 호출·매핑 테이블·if/else 분기 없음 — 2번 Group명과 4번 team_alias가 다르면 엉뚱한 팀으로 연결됩니다. ⚠️ 토큰은 access token만 허용(id_token은 401).

org-sso 대비: permission set → Cognito User Pool Group · aws sso login → llmgw-login · 근거: shared/reference/account-instance-setup.md · shared/patterns/lambda-handlers.md · shared/reference/litellm-admin-guide.md

배포 후 운영 · 개발자 온보딩 (Claude Code + Codex 클라이언트)

개발자 온보딩 — setup 한 번이면 클코·코덱스 동시 연결

🚀 한 번에 끝 (제로터치)

  • scripts/setup-developer.sh (Windows: .ps1 / py -3 gateway_auth.py setup) — outputs.json에서 게이트웨이 URL·모델 별칭·인증모드 자동 도출
  • ~/.claude/settings.json + ~/.codex/config.toml을 병합(백업 후 우리 키만 수정 — hooks·플러그인 보존)
  • 이후 로그인: org-sso aws sso login(IdC 세션 8h) / cognito llmgw-login(refresh 30d)

🧹 병합이 "지우는" 것 (실사고 2건)

  • 직접호출 잔재 CLAUDE_CODE_USE_BEDROCK·AWS_BEARER_TOKEN_BEDROCK 제거 — 남으면 두 인증 경로 충돌 → 403 "API key not valid"
  • 최상위 model(raw Bedrock ID) 제거 — 별칭보다 우선 적용돼 "team not allowed to access model"

⚙️ Codex 특칙 (실사고 2건)

  • auth는 command(실행파일)+args(배열) 분리 — Codex는 셸 없이 spawn, 합친 문자열은 401 "No api key"
  • web_search = "disabled" 필수 — Mantle이 web_search 툴 거부 → "high demand" 오표시 루프. 검색은 게이트웨이 AgentCore MCP로

🔑 인증 체감 — 개발자는 로그인만

  • 가상키는 1시간마다 자동 회전 — apiKeyHelper/auth.command가 만료 시 조용히 재발급, 개발자 무감
  • 재로그인 주기는 IdP 세션이 결정(키 duration과 무관) — 짧은 키 수명은 보안 창만 좁힘
  • 검증: healthcheck → /v1/models에서 별칭 목록 확인
