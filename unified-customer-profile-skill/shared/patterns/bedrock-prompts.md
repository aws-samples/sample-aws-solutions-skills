# Bedrock Prompts — AI Rule Generation & Improvement

Bedrock prompt patterns proven in existing demos. Use them as-is for rule generation/improvement.

## Model Selection (Runtime, Dynamic)

**Do not hard-code the model ID.** The user selects the model in the UI (the Select component in `AiRulesPage`), or a default is set via environment variable. For the catalog and pricing, see `shared/reference/aws-services.md`.

```typescript
// backend/lambdas/_shared/bedrock.ts
import { BedrockRuntimeClient, InvokeModelCommand } from '@aws-sdk/client-bedrock-runtime';

const br = new BedrockRuntimeClient({});

const DEFAULT_MODEL = process.env.BEDROCK_MODEL_ID ?? 'anthropic.claude-sonnet-4-20250514-v1:0';
const PERSONALIZATION_MODEL = process.env.PERSONALIZATION_MODEL_ID ?? 'anthropic.claude-haiku-4-5-20251001';

export async function invokeClaude(
  prompt: string,
  opts: { modelId?: string; maxTokens?: number; thinking?: 'adaptive' | 'enabled' | 'off' } = {}
) {
  const modelId = opts.modelId ?? DEFAULT_MODEL;

  // Opus 4.7 only supports adaptive thinking — guardrail
  let thinking: any = undefined;
  if (opts.thinking && opts.thinking !== 'off') {
    if (modelId.includes('opus-4-7')) thinking = { type: 'adaptive' };
    else thinking = { type: 'enabled', budget_tokens: 4000 };
  }

  const body = {
    anthropic_version: 'bedrock-2023-05-31',
    max_tokens: opts.maxTokens ?? 8192,
    messages: [{ role: 'user', content: prompt }],
    ...(thinking && { thinking }),
  };

  const r = await br.send(new InvokeModelCommand({
    modelId, body: JSON.stringify(body),
    contentType: 'application/json', accept: 'application/json',
  }));

  const parsed = JSON.parse(new TextDecoder().decode(r.body));
  return parsed.content?.find((c: any) => c.type === 'text')?.text ?? '';
}
```

## Prompt Composition Principles

The prompt sent to Bedrock is composed by combining the following sections:

```
[Explanation of how ER works] + [Actual data quality analysis] + [Schema information] +
[Current rules] + [User feedback] + [Current matching metrics] + [Unmatched pattern analysis] +
[Output format specification]
```

**Key**: include as much real data context in the prompt as possible to get accurate rules.

## 3. Advanced Rule Generation Prompt (Full Template)

```
Please recommend Entity Resolution Advanced Rule matching rules.

## How ER Advanced Rules Work

### Basic characteristics
- Advanced Rules support conditional fuzzy matching.
- Each rule is defined by a conditionString and can use various comparison functions.
- The relationship between rules is OR: if any one matches, the records are judged to be the same customer.
- Within a condition, AND, OR, and parentheses can be used to build compound conditions.

### Available comparison functions
- Exact(field): exact match
- Cosine(field, threshold): cosine similarity (0.0–1.0, effective for names)
- Levenshtein(field, threshold): edit distance (effective for phone/email typos)
- Soundex(field): phonetic-similarity matching (English names)
- DateDifference(field, days): date difference range

### Condition syntax
```
Exact(Email) AND Cosine(Name, 0.8)
(Exact(Phone) OR Levenshtein(Phone, 2)) AND Cosine(Name, 0.7)
Exact(LoyaltyNumber)
```

## The following is context auto-generated from actual data

{schemaSection}
{currentRulesSection}
{dataQualitySection}
{sampleSection}
{feedbackSection}
{metricsSection}
{unmatchedSection}

## Output format

Provide the recommended rules in the JSON format below:

```json
{
  "rules": [
    {
      "ruleName": "NameAndEmail",
      "conditionString": "Cosine(Name, 0.8) AND Exact(Email)",
      "description": "Name similar + email exact match",
      "estimatedPrecision": 0.95,
      "estimatedRecall": 0.7,
      "rationale": "Covers most records that have an email. Names use cosine to allow Korean-English variants."
    },
    {
      "ruleName": "NameAndPhone",
      "conditionString": "Cosine(Name, 0.7) AND Levenshtein(Phone, 2)",
      "description": "Name similar + phone similar",
      "estimatedPrecision": 0.9,
      "estimatedRecall": 0.6,
      "rationale": "Bypasses OTA relay email. Allows phone format differences (hyphen/country code)."
    }
  ],
  "reasoning": "Overall strategy explanation",
  "dataQualityAnalysis": "Summary of data characteristics analysis"
}
```

## Rule Design Guidelines

1. **Beware of empty fields**: fields with a high empty rate (>30%) are unsuitable as a standalone condition
   - If Email is 15% empty → an Email-only rule covers only 85% of records
   - If the name is empty → a DOB+Phone or DOB+Email combination is needed

2. **Cosine vs exact match**:
   - Name: Cosine(Name, 0.7–0.8) recommended (Korean-English mix, surname/given-name order differences)
   - Email: Exact(Email) recommended (a different domain means a different person)
   - Phone: Levenshtein(Phone, 2) recommended (format differences)

3. **Relay email handling**:
   - OTA relay (guest-xxx@booking.com) → the Email condition is meaningless
   - For this channel, fall back to Name+Phone

4. **Preventing false positives**:
   - "김민호" is very common in Korea → never use Name alone
   - Always combine Name + (Email OR Phone OR DOB)

5. **Do not affect existing matches**:
   - Adding a new rule must not break existing match groups
   - Add unmatched-coverage rules with OR or compose them as separate rules
```

## 4. Simple Rule Generation Prompt (Core Part)

```
Please recommend Entity Resolution Simple Rule matching rules.

## How ER Simple Rules Work (based on official documentation)

### Basic characteristics
- Simple Rules support only AND + ExactMatch. OR, parentheses, and fuzzy matching cannot be used.
- Each rule is defined by a matchingKeys array, and all specified fields must match exactly to be a match.
- Data normalization (applyNormalization) is enabled, so case, phone number format, address abbreviations, etc. are handled automatically.

### Waterfall matching (rule order matters)
- Rules are applied top to bottom in order (hierarchical waterfall).
- If rule 1 matches A↔B and rule 2 matches B↔C, then A, B, and C are transitively merged into a single group.
- Placing strict rules (many keys) first secures base matches with high precision, while looser rules (fewer keys) cover the rest.

### The essence of transitive merging
- Record A: LoyaltyNumber="L001", Email="a@gmail.com", Phone="01012345678"
- Record B: LoyaltyNumber="L001" (rule 1: LoyaltyNumber → A↔B match)
- Record C: Email="a@gmail.com" (rule 2: Email → A↔C match)
- Result: A, B, C all in the same group (transitively linked through A)

## Output format

```json
{
  "rules": [
    {
      "ruleName": "LoyaltyNumberMatch",
      "matchingKeys": ["loyaltynumber"],
      "rationale": "Unique identifier. Low empty rate and high uniqueness.",
      "estimatedPrecision": 0.99,
      "estimatedRecall": 0.4
    },
    {
      "ruleName": "NameAndEmail",
      "matchingKeys": ["firstname", "lastname", "email"],
      "rationale": "Name+email combination. Case handled by normalization.",
      "estimatedPrecision": 0.95,
      "estimatedRecall": 0.6
    }
  ],
  "ruleOrder": ["LoyaltyNumberMatch first (strictest)", "NameAndEmail next"]
}
```
```

## 5. User Validation Feedback Section

Feedback after the user validates matching results in HITL:

```
## User validation feedback (important — incorporate into rule improvement)

### Incorrect matches (False Positive) — 3 cases
Cases where the current rules incorrectly matched different customers as the same customer. The relevant rule must be strengthened (add conditions).
- Record A: {"firstname":"KIM","lastname":"MINHO","email":"minho1@gmail.com","phone":"01012345678"}
  Record B: {"firstname":"KIM","lastname":"MINHO","email":"minho2@gmail.com","phone":"01087654321"}
  → Same-name different person! Name alone is insufficient; Email or Phone must also match

### Missed matches (False Negative) — 2 cases
Cases where the current rules failed to match the same customer. Analyze the common field patterns of these pairs and add a new rule.
- Record A: {"firstname":"MINHO","lastname":"KIM","email":"minho@gmail.com","phone":""}
  Record B: {"firstname":"KIM","lastname":"MINHO","email":"minho@gmail.com","phone":"01012345678"}
  → First/last name swap! Solved by using Cosine(Name)
```

## 6. Unmatched Record Analysis Section

Comparison between records left unmatched after the ER run and their most similar group:

```
## Unmatched record analysis (important — add a rule that covers this pattern)
There are 15 unmatched records in total. Results of comparing each unmatched record with its most similar matched-group record:

Unmatched record: {"firstname":"","lastname":"","email":"minho@gmail.com","phone":"01012345678","sourcechannel":"OTA"}
Similar group record: {"firstname":"KIM","lastname":"MINHO","email":"minho@gmail.com","phone":"01012345678","sourcechannel":"WEB"}
Matching fields: email, phone
Mismatch cause: firstname: empty (original: KIM), lastname: empty (original: MINHO)
Empty fields: firstname, lastname — these fields are empty in this record, so using them in match conditions will not produce a match

Analyze the pattern above and add a rule that covers the unmatched records:
- An unmatched record's empty fields cannot be used as match conditions for that record. Build conditions using only non-empty fields.
- If the name is empty → a rule that matches using only DOB+Phone or DOB+Email, without Name/Cosine(Name)/Soundex(Name) conditions, is needed
- If names are swapped → use Cosine(Name) (instead of Exact(Name))
- Phone format differences → use Levenshtein(Phone,threshold)
- Email domain differences → use Levenshtein(Email,threshold)
```

## 7. Current Matching Metrics Section (Guardrail)

```
## Current matching results (important — do not worsen these metrics)
- Total records: 500
- Match groups (2+ records): 120
- Unmatched profiles (singletons): 80
- Suspect groups (exceeding channel count): 3

Goals for recommended rules:
1. Reduce unmatched profiles (improve recall) while
2. not increasing suspect groups (maintain precision)
3. Maintain or increase the number of existing match groups
4. Do not recommend rules that increase unmatched profiles beyond the current level
```

## Implementation Notes

### Prompt composition order
```typescript
function buildPrompt(...): string {
  // 1. How ER works (static — Simple/Advanced depending on rule type)
  // 2. Schema info (dynamically loaded from ER GetSchemaMapping)
  // 3. Current rules (if any — improvement mode)
  // 4. Data quality analysis (dynamically generated from S3 data)
  // 5. Sample records (10 extracted from S3)
  // 6. User validation feedback (loaded from DynamoDB — if any)
  // 7. Current matching metrics (computed from S3 ER results — if any)
  // 8. Unmatched pattern analysis (analyzed from S3 ER results — if any)
  // 9. Output format specification (JSON)
}
```
